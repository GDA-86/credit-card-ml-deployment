# Архитектура проекта: Credit Card ML Deployment

Данный документ описывает структуру и компоненты системы машинного обучения для классификации кредитных карт и её развертывания.

## 1. Технологический стек
*   **Язык:** Python 3.14.0
*   **Анализ и разработка:** Jupyter Notebooks (EDA, Feature Engineering)
*   **Веб-фреймворк:** Flask - логика в `api.py`
*   **Интерфейс:** HTML  templates/index.html
*   **Контейнеризация:** Docker & Docker Compose
*   **Модели:** Scikit-learn (артефакты `.pkl`)

## 2. Структура проекта
Проект организован по модульному принципу:

*   `app/` — Серверная часть приложения.
    *   `api.py` — Точка входа для API запросов.
    *   `my_transformers.py` — Пользовательские классы для трансформации данных.
    *   `templates/` — HTML-шаблоны для веб-интерфейса.
*   `dev/` — Среда исследования (Research & Development).
    *   Содержит ноутбуки для EDA, генерации признаков и разработки моделей.
    *   `UCI_Credit_Card.csv` — Исходный набор данных для экспериментов.
*   `models/` — Управление моделями.
    *   `model_trian.py` — Скрипт для обучения модели.
    *   `model_v1.pkl`, `model_v2.pkl` — Версионированные веса моделей.
*   `docker/` — Конфигурационные файлы сборки образа (`Dockerfile`).
*   `tests/` — Папка с тестами для проверки корректности работы API.

## 3. Процесс разработки и деплоя 
1.  **Исследование:** Анализ данных и создание модели в папке `dev/`.
2.  **Обучение:** Запуск `model_trian.py` для генерации актуального `.pkl` файла.
3.  **Сборка:** Создание Docker-образа на основе `docker/Dockerfile`.
4.  **Тестирование:** Проверка API через `tests/test_api.py`.
5.  **Публикация:** Загрузка образа в публичный Docker Hub для дальнейшего использования.
6.  **Эксперименты:** План проведения A/B тестирования новых моделей описан в `ab_test_plan.md`.



## Часть 3. Сервисная архитектура

### 3.1. Выбор архитектуры: Монолит

Для реализации проекта по предсказанию банкротства была выбрана **монолитная архитектура**. Н

1.  **Простота разработки и деплоя**
    Поскольку проект решает одну конкретную задачу, разделение его на отдельные сервисы увеличивает трудозатраты, которые целесообразно потратить на развитие самой модели. Один сервис проще упаковать в Docker и развернуть на одной виртуальной машине.

2.  **Развитие и модернизация модели**
    Значительно увеличивается скорость по выпуску новой модели - так как ге нало переписывать все взаимодействия, достаточно обновить модель и образ.

3.  **Экономия производительности**
    Микросервисная архитектура требует больших вычислительных мощностей для запуска множества контейнеров, API-шлюзов и систем мониторинга. Монолитная структура потребляет гораздо меньше оперативной памяти.


### Использование брокера очередей (RabbitMQ)

1. **Асинхронная обработка предсказаний:**
   Если пользователю нужно загрузить файл с тысячей строк  одновременно, синхронное ожидание ответа (HTTP-запрос) приведет к тайм-ауту. В этом случае API принимает файл, сохраняет его и отправляет задачу в RabbitMQ. Отдельный воркер (Worker) забирает задачу из очереди, выполняет предсказания в фоновом режиме и сохраняет результат в базу данных

2. **Планирование загргузки ресурсов:**
   Брокер выступает буфером. Если в систему одновременно придет слишком много запросов, они не «уронят» сервис предсказаний, а просто выстроятся в очередь и будут обработаны по мере освобождения ресурсов.



### 3.2. Обзор инструментов MLOps: DVC и MLflow

#### 1. DVC (Data Version Control) — Контроль версий данных


* DVC решает проблему версионирования тяжелых объектов (датасетов, весов моделей), которые невозможно эффективно хранить в Git.

*   **Назначение:** Управление версиями данных и артефактов, а также создание воспроизводимых конвейеров (pipelines) обработки данных.
*   **Роль в жизненном цикле:**
    *   **Data Engineering:** Фиксирует «слепки» данных на этапах очистки и трансформации
    *   **Воспроизводимость:** Гарантирует, что конкретная версия кода всегда будет запущена на соответствующей ей версии данных
    *   **Сотрудничество:** Позволяет команде синхронизировать состояние данных между локальными машинами


#### 2. MLflow — Отслеживание экспериментов

MLflow выступает в роли единого журнала регистрации всех действий исследователя данных (Data Scientist).

*   **Назначение:** Логирование параметров, метрик и артефактов обучения, а также управление реестром моделей
*   **Роль в жизненном цикле:**
    *   **Экспериментирование :** Сравнение различных итераций обучения по ключевым метрикам и гиперпараметрам
    *   **Управление моделями :** Централизованное хранилище
    *   **Стандартизация :** Упаковка кода в формат, пригодный для повторного запуска в любой среде



### 3.3. Бизнес-метрики проекта

Технические метрики (F1-score, Precision) позволяют оценить качество работы алгоритма, но не дают прямого ответа на вопрос о прибыльности решения. Для оценки эффективности модели со стороны бизнеса предлагается использовать следующие показатели:

* Уровень просрочки в деньгах 

* Динамика кол-ва рискованных клиентов 

* Диамика общей суммы убытока 





## Часть 4 Концепция A/B-тестирования 

## 1. Постановка эксперимента

*   **Объект тестирования:** Прогнозирование дефолта клиентов по кредитным картам.
*   **Группы:**
    *   **Контрольная (v1):** Текущая модель - GradientBoostingClassifier
    *   **Тестовая (v2):** Новая модель Logistic Regression
*   **Разделение трафика:** 
    *   Случайное распределение **50/50** на основе хеша `ID` клиента 
    *   Это гарантирует, что один и тот же клиент при повторных запросах всегда попадает в одну и ту же группу
*   **Продолжительность:** 
    *   **Технический тест:** 1–2 недели  
    *   **Бизнес-тест:** Минимум 1 полный платежный цикл: 3–6 месяцев для фиксации реальных фактов банкротства

## 2. Метрики и обоснование

### Основная метрика: F1-score 
*   **Почему:** F1-score является гармоническим средним между точностью и полнотой, что позволяет оценить качество модели без искажений, вносимых преобладающим классом положительных клиентов

### Дополнительная метрика: Recall
*   **Почему:** Для бизнеса  важно минимизировать пропуск банкротных клиентов (False Negatives). Ошибка пропуска теоретического банкрота банку дороже, чем ошибочный отказ надежному клиенту 
*   **Обоснование выбора:** следим за Recall, чтобы убедиться, что новая модель не стала слишком «осторожной»

## 3. Статистический анализ

### Выбор метода: U-критерий Манна-Уитни: 
U-критерий Манна-Уитни (Mann-Whitney U Test): Сравнивает медианы двух независимых выборок и подходит для нашего случая с ненормальным распрделеним 

### Доверительные интервалы
*   Строится **95% центральный доверительный интервал** для разности метрик между группами.
*   Если интервал лежит полностью выше нуля — разница статистически значима

### Критерий успешности
* Тест прошел успешно, если выполнены три условия:
   * Результат не случаен: Статистика подтверждает, что новая модель (v2) действительно лучше старой, а не просто «повезло» на конкретных данных
   * Эффект заметен для бизнеса: Улучшение качества (F1-score) достаточно значимое, чтобы оправдать затраты на внедрение новой модели
   * Нет критических ошибок: Новая модель не стала чаще пропускать реальных должников (Recall не упал) по сравнению с текущей версией
